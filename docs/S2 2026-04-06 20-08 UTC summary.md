## What was fixed in the last hours

### 1) Runtime execution unlock

The runtime gate for multi-token live entry was unlocked by aligning the risk/execution path with the effective resolved bot settings rather than letting old shared safe-mode assumptions block valid entries.

This removed blockers such as:

- trading disabled
- BTC-only symbol allowlist
- low max leverage assumptions preventing active bot symbols from becoming actionable

### 2) Missing price-tick execution blocker

Execution was previously failing at sizing with:

- `No stored price tick available for sizing`

This was fixed by adding a fallback to a live Bybit market ticker price when no stored internal symbol tick exists.

This removed a fragile dependency on pre-populated tick storage.

### 3) Live sizing basis fix

A major sizing bug was identified and fixed.

The intended sizing rule is:

- `effective account balance = min(actual account balance, 5000 USD)`
- `trade notional = effective account balance × % of account to use × leverage`

The bug was that S2 had been treating `5000` as the active sizing base instead of as a cap.

This produced absurd order quantities and caused Bybit insufficient-balance rejections.

After the fix, live sizing correctly used the real subaccount balance capped at 5000.

### 4) Webhook/dashboard DB source-of-truth alignment

The dashboard and webhook runtime were aligned onto the same explicit DB path so recent runtime activity can be represented from one source of truth.

This was necessary because mobile/dashboard observability was lagging behind the live webhook/order behavior.

### 5) Mobile operator page improvements

The mobile page now correctly reflects:

- enabled bot set
- symbol mapping
- balances
- current trade state

Additional recent monitoring blocks were added for:

- last trade signal
- last order attempt
- last failed execution reason

However, one small remaining inconsistency still appears to exist in the mobile summary block’s DB-read path, described below.
